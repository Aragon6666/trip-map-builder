# AGENTS.md

## 仓库用途

这个仓库是一个可复用的旅游规划工作区。它把用户的目的地、约束、偏好、调研记录和地点坐标，整理成适合手机打开的交互式旅行地图页。

仓库分两层：

- `references/`：通用规划和调研方法。
- `trips/<trip_id>/`：每次旅行一个独立目录，保存该次旅行的需求、调研、行程、结构化数据和最终 HTML 地图。

## 当前结构

```text
assets/template.html              # 通用 Leaflet 地图模板
references/                       # 通用规划/调研工作流
memory/traveler_profile.md        # 长期可复用旅行偏好
trips/YYMM_destination/            # 每次旅行一个目录
  requirements.md                 # 本次旅行个性化需求，短期记忆
  research.md                     # 调研记录和待复核事项
  itinerary.md                    # 人类可读每日行程
  trip_data.js                    # HOTEL + DAYS 结构化数据
  index.html                      # 最终交互式旅行地图
index.html                        # Vercel/static hosting 的旅行列表首页
README_SELF.md                    # 面向使用者和开发者的快速说明
```

旅行目录使用简短命名，例如 `trips/2606_wuhan/`。

## 规划新行程时

如果用户只给了一个目的地，不要立刻开始规划。先一次性问几个必要问题，等待用户补充后再开始。

缺失时优先询问：

- 日期、出发城市、抵达/离开时间、交通方式
- 是否已订酒店；如未订，询问酒店位置/预算/卫生/交通等要求
- 人数、关系/场景、旅行节奏
- 必去地点、不想去的地点、兴趣主题和特殊偏好
- 餐饮偏好、吃辣程度、预算、排队接受度、忌口/过敏
- 如有必要，询问最终希望本地 HTML 使用还是部署到 Vercel

用户补全信息后，再按下面流程执行。

## 旅行规划流程

1. 读取 `memory/traveler_profile.md`，复用长期旅行偏好。
2. 创建或更新 `trips/<trip_id>/requirements.md`，记录本次旅行的个性化需求。
3. 调研目的地区域、酒店、景点、餐饮、交通、票务、开放时间、天气和风险点。
4. 把有效信息和待复核事项写入 `research.md`。
5. 在 `itinerary.md` 中写出轻松、按区域组织的每日行程。
6. 在 `trip_data.js` 中整理结构化 `HOTEL`、`DAYS`、`WEATHER` 和 `PACKING` 数据。
7. 基于 `assets/template.html` 生成或更新 `trips/<trip_id>/index.html`。
8. 更新根目录 `index.html`，让新旅行出现在旅行列表首页。
9. 用本地浏览器验证旅行页能正常显示地图、日期 tab、地点卡片、天气说明和行李清单。

优先做顺路、舒服、有留白的路线，不要为了塞满景点牺牲体验。通常一天一个主区域更稳。需要主动删掉高风险或过远的点，并说明原因。

## 地图展示规则

旅行地图页应保持统一交互风格：

- 地图点按地点类型使用固定颜色，而不是只按日期上色。默认类型包括：住宿 `hotel`、景点 `spot`、室内备选 `indoor`、餐馆 `food`、茶饮/咖啡 `tea`、交通 `transport`。
- 当日地图应按详细行程顺序显示数字标号，并用路线线段或箭头表达大致移动方向，让用户只看地图也能理解当天路线。
- 下方行程卡片必须和地图点建立联动：点击卡片时，高亮对应地图点并打开弹窗；点击地图点时，应能高亮或定位到对应卡片。
- 备选点可以显示在详细行程中，但不要混进主路线编号；除非当天临时改用该点。
- 图例要在总览或每日页可见，帮助用户理解颜色。

## 天气规则

默认使用 Open-Meteo Forecast API 作为天气信源。出行日期在未来 14 天内时，可以写入逐日天气预报，并根据降雨、温度、风等因素调整室内/室外安排。超过 14 天时，不要虚编天气，只提醒用户出发前 7-10 天刷新。

每次旅行页面最后应有一个“行李”tab，展示天气影响和待办式行李清单。固定包含电脑、充电器、鼠标、iPad、刮胡刀、指甲钳、相机；天气或目的地特有物品应加粗或单独列出。

## 交付给用户时

完成旅行规划后，必须告诉用户两种使用方式：

1. 直接打开生成的 HTML，例如 `trips/<trip_id>/index.html` 或本地服务 URL。
2. 部署/查看 Vercel。根目录 `index.html` 是旅行列表页，每次旅行可通过 `/trips/<trip_id>/` 访问。

同时提醒用户出发前仍需复核的事项，例如官方开放时间、博物馆预约、门票规则、轮渡/交通班次、天气和最终酒店地址。

## 开发规则

- 通用工作流写在 `references/`。
- 长期偏好写在 `memory/traveler_profile.md`。
- 单次旅行的需求、调研和决策写在对应 `trips/<trip_id>/` 目录。
- 不保存证件、订单号、完整聊天记录、私密标识等敏感信息。
- 不要把某个目的地硬编码进通用模板；目的地相关内容应放进具体旅行目录。
- 后续如果新增脚本，优先使用 `trip_data.js` 或 JSON 这类确定性输入，让 HTML 可复现生成。

## 维护本文件

当代码层面或工作流层面的变化影响以下内容时，需要同步更新 `AGENTS.md`：

- 仓库目录结构
- 旅游规划流程
- 每次旅行的输出文件约定
- 地图生成方式
- 部署说明
- agent 使用这个仓库的规则和预期

仅新增一个目的地，或用户个人旅行偏好变化，不需要更新本文件；除非这些变化同时改变了底层工作流或代码行为。
